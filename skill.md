# React Engineering Profile

## Role

Senior React Developer

## Experience

5+ years of professional experience building scalable, production-grade React applications.

---

## Engineering Identity

I prioritize:

- Clean and readable code
- Simple and maintainable architecture
- Long-term scalability
- Performance-conscious implementation
- Clear separation of concerns
- Business-aligned technical decisions

---

## Code Philosophy

### Readability > Cleverness

- Use descriptive variable and function names.
- Code should be understandable by reading it once.
- Avoid unnecessary comments by writing self-explanatory code.
- Avoid deeply nested logic.
- Prefer early returns over multiple `if-else` chains.

### Simplicity First

- Avoid over-engineering.
- Do not introduce abstractions without clear need.
- Build for current requirements while allowing reasonable extensibility.
- Avoid premature optimization.

### Maintainability

- Keep functions small and single-responsibility.
- Separate business logic from UI logic.
- Keep components modular and reusable.
- Prefer composition over complex inheritance patterns.
- Ensure future developers can extend features easily.

---

## Component Architecture

- Break large components into smaller reusable components.
- Maintain smart vs presentational separation when appropriate.
- Avoid monolithic files.
- Design components for reuse, not duplication.
- Keep props clean and predictable.

---

## Performance Mindset

- Prevent unnecessary re-renders.
- Use `React.memo`, `useMemo`, and `useCallback` intentionally (not blindly).
- Avoid prop drilling where architecture patterns solve it better.
- Apply lazy loading and code splitting when needed.
- Keep bundle size and rendering performance in mind.

---

## Project Structure Principles

- Prefer feature-based structure when appropriate.
- Avoid creating unnecessary files.
- Every file must have a clear responsibility.
- Keep directory structure predictable and scalable.
- No “rubbish files” or unused abstractions.

---

## Guard Rails

- Do not over-abstract.
- Do not add libraries without strong justification.
- Do not create generic utilities unless reuse is proven.
- Do not introduce complexity to appear advanced.
- Keep solutions aligned with product goals.

---

## Coding Standards

- Consistent naming conventions.
- Pure and testable logic.
- Clear state management patterns.
- Strong linting and formatting discipline.
- Prefer explicitness over hidden magic.

---

## System Thinking

- Think in flows, not just components.
- Optimize for team scalability.
- Design for predictable growth.
- Balance delivery speed with code quality.
- Focus on long-term maintainability.

---

## Output Expectation (For LLM Context)

When generating React code for me:

- Prefer simple, readable solutions.
- Avoid nested conditional complexity.
- Break logic into small reusable functions.
- Keep components modular.
- Avoid unnecessary abstraction layers.
- Keep performance in mind.
- Do not over-engineer.
- Use meaningful naming.
- Structure code cleanly and predictably.

---

## Core Principle

Great frontend engineering is not about complexity.

It is about:

- Clarity
- Predictability
- Maintainability
- Performance
- Scalability
